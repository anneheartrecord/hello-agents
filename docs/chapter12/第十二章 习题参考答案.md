# 第十二章 习题参考答案

> 参考答案仅供参考。本章最重要的不是背指标，而是建立一个清晰判断：Agent 评估一定是“任务目标、执行过程、成本约束、业务风险”四者共同作用的结果。

## 1. 评估基准、客服评估设计与三大挑战

### BFCL 与 GAIA 的区别

`BFCL` 主要评估工具调用能力，核心关注点是：

- 是否选对函数
- 参数是否正确
- 多工具调用是否合理

`GAIA` 更偏综合智能体能力，关注：

- 多步任务完成
- 外部信息检索
- 复杂问题求解
- 最终答案质量

BFCL 用 AST 匹配，是因为工具调用更像“结构化程序片段”；GAIA 用准精确匹配，是因为最终答案常有多种表述形式。

| 方法 | 优点 | 缺点 |
|------|------|------|
| AST 匹配 | 结构敏感，适合函数调用验证 | 对语义等价但结构不同的情况可能误判 |
| 准精确匹配 | 适合答案多样性，规则较直观 | 对复杂语义等价判断能力有限 |

### 智能客服系统的评估指标设计

可按能力拆分：

1. 意图理解准确性：分类准确率、召回率、混淆矩阵。
2. API 调用正确性：工具调用成功率、参数正确率、执行顺序正确率。
3. 回答友好性和专业性：LLM Judge + 人工抽检 + 术语规范检查。
4. 异常处理鲁棒性：对脏输入、超时、缺失信息、恶意输入的压力测试。

原则是：能自动化的尽量自动化，主观维度再引入 LLM Judge 和人工复核。

### 三大挑战及解决思路

1. 输出不确定性：
   可采用多次采样、分布式统计和稳定性评估，而不是只看单次结果。

2. 评估标准多样性：
   可建立分层指标体系，把正确性、效率、安全性、体验分开打分。

3. 评估成本高昂：
   可采用分层评估，日常跑轻量集，发布前跑重型集。

局限性在于：成本能下降，但很难做到“零成本且高覆盖”。

## 2. BFCL：AST 匹配、边界样本与评估器扩展

### 为什么 AST 匹配优于字符串匹配

字符串匹配太脆弱，例如：

- 参数顺序变化
- 空格、换行、引号风格不同
- 等价表达但表面文本不同

AST 匹配更关注结构，因此更适合判断“调用语义是否基本一致”。

但 AST 也可能误判：

- 假阳性：结构相似，但关键默认参数语义不同。
- 假阴性：结构不同，但通过变量展开后其实等价。

改进方向：引入 schema 校验、参数归一化和执行前模拟验证。

### 为四类样本设计新边界测试

- `simple`：缺失可选参数、单位不明确、别名函数调用。
- `multiple`：两个工具间存在弱依赖但顺序容易错。
- `parallel`：可并发任务中混入一个必须串行的子任务。
- `irrelevance`：用户提到工具关键词，但真实意图并不需要调用。

这类样本更能暴露模型“看见关键词就乱调工具”的问题。

### 扩展 BFCL 评估器的方向

新增三类能力：

- 顺序评估：检查依赖型调用是否按正确顺序执行。
- 效率评估：是否用了最少或接近最少的调用次数。
- 错误分析报告：按错误类型聚合，例如选错工具、参数错、顺序错、过度调用。

这样评估结果不只是一个分数，而是能指导优化的诊断报告。

## 3. GAIA：难度级别、答案匹配与自定义评估集

### Level 1/2/3 的差异，以及 Level 4 该长什么样

一个合理理解是：

- `Level 1`：任务较短、依赖能力少、答案路径简单。
- `Level 2`：需要更多检索、整合和中等复杂推理。
- `Level 3`：多步骤更长、依赖更多工具和跨源整合。

如果设计 `Level 4`，我会加入：

- 更长时程任务
- 动态环境变化
- 多工具依赖链
- 对安全性、成本和计划能力的联合要求

### 准精确匹配的不足与更智能方案

准精确匹配通常会做数值归一化、大小写归一化、格式清洗，因此能识别 `42`、`42.0`、`四十二` 这类简单等价。

但它不够处理：

- 复杂改写
- 语义等价但表述跨度很大的答案
- 多字段复合答案

更智能方案可分三层：

1. 规则归一化。
2. 结构化字段比对。
3. LLM Judge 判断语义等价，并要求给出解释。

### 自定义 GAIA 评估集的设计建议

以“金融领域”为例，可以设计 10 个问题，覆盖：

- Level 1：读取单份财报中的显式指标。
- Level 2：结合两份报告做同比分析。
- Level 3：综合财报、新闻、行业数据做风险判断。

每题应同时定义：

- 标准答案
- 可接受变体
- 所需工具
- 评分规则

这样才能保证评估结果可重复、可解释。

## 4. LLM Judge 与多评委评估

### LLM Judge 的优势与局限

优势：

- 能评估难以规则化的质量维度。
- 对创意、表达、逻辑完整性等维度更灵活。
- 更适合开放式生成任务。

局限：

- 可能偏好某些文风。
- 对长度可能存在偏见。
- 不同 Judge 之间一致性未必稳定。
- 若评分标准写得不清楚，结果会漂移。

### 三种场景的评分标准设计

1. 代码生成：
   正确性 40%，可读性 20%，效率 20%，安全性 20%。

2. 创意写作：
   创意性 30%，叙事完整性 25%，语言表现 25%，情感感染力 20%。

3. 技术文档：
   准确性 35%，结构清晰度 25%，覆盖完整性 25%，术语一致性 15%。

每一项最好都配示例锚点，避免 Judge 漂移。

### 多评委评估系统怎么做

可采用 3-5 个不同模型，流程如下：

1. 使用统一 rubric 独立评分。
2. 对分数做标准化。
3. 采用中位数或截尾平均聚合。
4. 若分歧过大，触发复审或仲裁 Judge。

异常评分可通过以下规则识别：

- 与其他评委偏差过大
- 理由明显空泛
- 与历史评分风格严重不一致

## 5. 分层评估、持续评估与报告生成

### 分层评估策略

- 快速评估：小样本、关键路径、低成本，用于日常迭代。
- 标准评估：覆盖主流程、核心边界场景，用于发版前。
- 全面评估：加入压力测试、长任务、人工评估和对抗样本，用于重大更新。

评估流程可统一为：准备数据 -> 运行评估 -> 聚合指标 -> 生成报告 -> 触发门禁或告警。

### 持续评估系统设计

系统组件可包括：

- 定时调度器
- 评估执行器
- 指标存储
- 趋势分析面板
- 告警系统

告警规则示例：

- 连续两次核心指标下降超过阈值
- 工具调用失败率突然升高
- 某个关键任务从通过变为失败

### 面向不同受众的报告生成系统

开发者报告应突出：

- 失败样本
- 错误类型分布
- 工具调用链路
- 版本间回归对比

产品经理报告应突出：

- 任务完成率
- 用户关键路径表现
- 成本变化
- 风险点和发布建议

用户报告应尽量简化为：

- 核心能力是否提升
- 哪些场景表现更稳定
- 用图表或等级而不是复杂技术术语展示

这说明评估不是只为模型团队服务，而是要支撑产品决策和外部沟通。
