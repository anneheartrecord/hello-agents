# 第九章 习题参考答案

> 参考答案仅供参考。本章真正的重点，是把“上下文”看成一种会腐蚀、会膨胀、需要治理的系统资源，而不是无限堆叠给模型的原材料。

## 1. 上下文工程与提示工程的边界

### 什么是 context rot，为什么长窗口不代表可以无脑塞满

`context rot` 可以理解为：随着上下文越来越长，真正有价值的信息被噪声、过期信息、重复片段、低相关内容不断稀释，模型虽然“看到了更多”，但决策质量反而下降。

即使模型支持 100K 甚至 200K 上下文，仍然要谨慎管理，原因包括：

- 长上下文不是免费资源，成本和延迟会上升。
- 模型对远距离关键信息的利用并不总是稳定。
- 冗余信息会干扰注意力分配。
- 越长的上下文，越容易混入过期、冲突、无关内容。

所以重点不是“能放多少”，而是“应该放什么”。

### 50 个文件代码审查：一次性全塞 vs JIT 检索

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 一次性全塞 | 全局视图完整，适合小仓库 | token 成本高，噪声大，更新不灵活 | 文件少、关系强、一次性整体分析 |
| JIT 按需检索 | 成本低，相关性高，可渐进探索 | 初始全局视图较弱，依赖检索质量 | 中大型代码库、交互式审查 |

对 50 个文件的仓库，我更推荐 `JIT + 少量全局摘要`：先给目录结构、关键模块摘要，再按问题需要拉具体文件。

### “过度硬编码”与“过于空泛”的例子

过度硬编码例子：

- “必须分 7 段回答，每段 80-100 字，第二段必须先夸赞用户，再输出 3 个建议。”

问题：把格式控制到过细，模型没有根据任务调整表达的空间。

过于空泛例子：

- “你是一个很棒的 AI 助手，请尽量帮助用户。”

问题：缺少边界、质量标准和约束，模型很难稳定执行。

平衡点通常是：明确目标、边界和关键格式，但不要把每一步都写死。

## 2. GSSC 流水线失效分析与改进

### 某个阶段失效会造成什么后果

GSSC 四阶段分别承担不同职责，因此某一层失效会产生不同后果：

- `Gather` 失效：信息源不全，后续再精细也无从补救。
- `Select` 失效：把无关信息选进来，噪声大幅增加。
- `Structure` 失效：信息虽对，但组织混乱，模型难以高效利用。
- `Compress` 失效：过度摘要导致关键约束和细节丢失。

最终表现可能是：答非所问、遗漏关键条件、重复搜索、工具调用错误、长任务中途偏航。

### “上下文质量评估”功能怎么做

可以在每次构建完上下文后打一个质量报告，至少包含三类指标：

- 信息密度：有效 token / 总 token。
- 相关性：上下文内容与当前任务 embedding 相似度。
- 完整性：任务所需关键槽位是否齐全，例如目标、约束、当前状态、历史决策。

输出示例：

```json
{
  "density": 0.71,
  "relevance": 0.84,
  "completeness": 0.76,
  "suggestions": [
    "删除重复日志片段",
    "补充最近一次失败原因",
    "压缩过长的终端输出"
  ]
}
```

### 何时截断或滑动窗口比 LLM 摘要更合适

当内容高度结构化或对原文精确性要求极高时，简单截断反而更可靠，例如：

- 错误堆栈
- 配置文件差异
- 日志尾部最新事件
- 表格型数据

一个混合压缩策略可以这样做：

- 对结构化原文用截断或滑动窗口保留原貌。
- 对长篇说明文用 LLM 摘要。
- 对关键约束单独抽槽，不参与自由摘要。

这样能兼顾信息保真和长度控制。

## 3. NoteTool、TerminalTool 与智能重构助手

### 自动整理笔记机制

可将临时笔记作为“待处理缓存层”，当数量或体积达到阈值时触发整理：

1. 先聚类相似临时笔记。
2. 提取高频主题、未完成事项、风险项。
3. 将长期有效的信息提升为任务笔记或项目笔记。
4. 对重复、过期、无价值内容归档或删除。

升级规则示例：

- 与当前任务直接相关且未来仍需执行 -> 任务笔记
- 跨多个任务共用的约束或架构信息 -> 项目笔记

### 当前安全机制是否足够，审批流程怎么加

路径验证、命令白名单、权限检查是必要基础，但对敏感操作还不够。因为风险不仅在“命令长什么样”，还在“命令作用于什么对象、后果是否可逆”。

更稳妥的审批流程：

1. Agent 先生成操作计划和影响说明。
2. 系统对命令做风险分级，例如低、中、高。
3. 高风险操作必须进入人工审批界面。
4. 审批后生成一次性授权 token。
5. 执行后记录审计日志与结果快照。

### 智能代码重构助手工作流

```text
读取代码库结构
  -> 生成重构计划
  -> NoteTool 记录阶段目标与风险
  -> TerminalTool 执行代码搜索/修改/测试
  -> NoteTool 记录结果与问题
  -> 若失败则回退到计划修订
  -> 若通过则推进下一子任务
```

这个流程的关键，不是让 Agent 直接改代码，而是让“计划、执行、验证、记录”形成闭环。

## 4. 长时程任务中的分层上下文管理

### 三层如何协调

三层更合理的职责划分是：

- `TerminalTool`：即时事实层，负责当前文件、命令输出、实时状态。
- `MemoryTool`：会话状态层，负责当前任务中的近期结论和中间决策。
- `NoteTool`：持久知识层，负责跨轮次、跨中断仍需保留的信息。

放错层会带来问题：

- 把临时终端输出长期存入笔记，会造成污染。
- 把长期约束只放在会话记忆，一旦中断就丢失。

因此需要“信息升级规则”，让内容只在必要时向上层沉淀。

### 断点续传机制怎么设计

笔记中至少要记录：

- 当前任务 ID
- 已完成子任务
- 未完成子任务
- 当前依赖状态
- 最近一次失败原因
- 关键文件和修改位置
- 下一步明确动作

恢复时可执行：

1. 读取任务快照。
2. 用 `TerminalTool` 验证关键状态是否仍成立。
3. 若状态漂移，则先做差异修正。
4. 确认后从“下一步动作”继续。

### 任务依赖管理系统

可将任务建模为 DAG：

- 节点表示子任务
- 边表示依赖关系
- 状态包括 `pending / running / blocked / done / failed`

与 `NoteTool` 集成时，可让每个任务节点自动映射到一条结构化笔记记录，便于恢复与审计。

## 5. 渐进式披露的价值、风险与适用边界

### 一个具体场景：复杂问题调试

以线上故障调试为例，渐进式披露流程可以是：

1. 先看错误摘要。
2. 再拉相关日志片段。
3. 再读对应模块代码。
4. 再查看配置和部署差异。
5. 最后形成修复假设。

这样做的好处是：每一步新信息都会收缩搜索空间，而不是一开始就把所有日志和代码一次性塞满上下文。

### 如何避免低效探索

可设计“探索引导”机制：

- 优先访问信息增益最高的来源。
- 当连续两步没有新增有效信息时，强制切换探索方向。
- 为每条探索路径设置预算，避免在枝节上无限下钻。
- 使用元认知提示，让模型先回答“我下一步为什么查这个”。

### 渐进式披露 vs 一次性加载的适用任务

渐进式披露更适合：

- 调试和故障定位
- 大型代码库分析
- 多步骤研究与资料收集

一次性加载更适合：

- 小规模短文改写
- 题目和材料都很短的问答
- 高度依赖全局一致性的短上下文任务

关键不是“哪种方法永远更好”，而是任务规模和信息分布决定策略。
