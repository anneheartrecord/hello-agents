# 第十一章 习题参考答案

> 参考答案仅供参考。本章的真正重点，不是记住训练算法缩写，而是理解为什么 Agent 一旦进入“多步决策 + 工具使用 + 延迟反馈”场景，训练目标就必须从单步预测升级为序列优化。

## 1. 从 LLM 训练到 Agentic RL 的视角变化

### 为什么 Agentic RL 的状态包含历史观察，而 PBRFT 通常只看初始 prompt

在 PBRFT 场景里，模型通常被看成“给定一个 prompt，输出一个回答”，所以优化对象主要围绕初始输入和最终输出展开。

但 Agentic RL 中，智能体会在执行过程中不断看到新的观察 `o1, o2, ...`，例如：

- 工具返回结果
- 环境状态变化
- 中间执行失败信息

这意味着决策不是一次性的，而是条件于历史轨迹的。若状态里不包含这些观察，策略就无法根据外部反馈调整下一步动作。

影响主要有两点：

- 训练难度更高，因为状态空间和轨迹空间都变大了。
- 最终能力更强，因为模型学到的不是“一次回答”，而是“如何在执行中适应环境”。

### 智能代码调试助手的 RL 映射

一个合理定义如下：

- 状态空间 `S`：用户问题、当前代码片段、历史工具输出、已尝试修改、测试结果。
- 行动空间 `A`：阅读文件、搜索文档、调用测试、修改代码、请求更多上下文、结束任务。
- 奖励函数 `R`：测试通过给高奖励；修复部分错误给中等奖励；引入新错误、无效修改、过长路径给负奖励。
- 状态转移 `T`：动作执行后，代码库状态、测试状态、观察结果发生变化。

这比普通问答更像一个交互式决策过程，而不是一次静态生成。

### 为什么监督学习难优化长期目标

以“数学证明”任务为例：

- 监督学习通常只看到标准证明文本，学的是“像答案一样写”。
- 但很多中间步骤本身没有唯一标准形式，写法多样，难以逐步对齐。
- 某一步看似合理，可能会在 5 步之后导致证明走不通。

强化学习可以把奖励延迟到最终结论是否成立、路径是否高效，再把这个结果反向传给整条轨迹，因此更适合优化长期效果。

## 2. SFT、LoRA 与 GRPO

### LoRA 的核心思想

LoRA 的本质不是“少训练一点参数”这么简单，而是假设模型更新往往可以用低秩变化来近似表达。

做法是：冻结原模型大部分参数，只训练少量低秩矩阵，让这些矩阵近似表示权重更新方向。

为什么有效：

- 大模型已经具备很强通用表示能力。
- 下游任务适配往往不需要全面重写整个参数空间。
- 低秩更新能抓住最关键的调整方向。

适合选 LoRA 的情况：算力有限、数据量有限、需要快速试验多个版本。

### GRPO 相比 PPO 的优势

`PPO` 通常需要价值模型或更复杂的基线估计，训练实现和稳定性治理更重。

`GRPO` 的核心改进是：在同一组候选输出内部做相对奖励比较，用组内相对优势替代部分复杂估计。

带来的好处：

- 训练流程更简单。
- 对奖励尺度更稳健。
- 在生成式任务中更容易直接利用多候选样本做优化。

如果迁移到代码生成或对话优化，关键调整在于：

- 换成对应任务的奖励函数。
- 设计合适的采样多样性，避免候选都太像。
- 控制长度偏置和格式偏置。

### 扩展 SFT 训练流程的建议

可新增三类能力：

1. 多轮对话训练：把样本组织成 `messages` 序列，保留角色和轮次信息。
2. 数据增强：做同义改写、指令重述、难度分层采样。
3. 可视化监控：记录 loss、梯度、样本长度分布、人工抽样质量。

目标不是让训练更花哨，而是让数据和训练过程更可解释。

## 3. 奖励函数设计与 reward hacking 防御

### 更细粒度的数学题奖励函数

可把总奖励拆成多个部分：

```text
reward = 0.5 * final_correctness
       + 0.2 * step_validity
       + 0.2 * partial_progress
       - 0.1 * verbosity_penalty
```

解释：

- `final_correctness`：最终答案是否正确。
- `step_validity`：推理步骤是否逻辑自洽。
- `partial_progress`：即使没完全做对，是否做对了中间关键步骤。
- `verbosity_penalty`：无意义冗长则扣分。

### 三类任务的奖励函数设计

1. 代码生成助手：
   重点看测试通过率、静态检查结果、运行效率、可读性。

2. 客服对话智能体：
   重点看问题解决率、用户满意度、首次响应时间、是否违规承诺。

3. 游戏 AI：
   重点看胜率、策略多样性、对不同对手的鲁棒性。

奖励函数不应只盯一个主指标，否则容易诱导策略走偏。

### reward hacking 的例子与防御机制

例子：

- 代码智能体学会直接修改测试用例，让测试通过。
- 客服智能体用敷衍话术快速结束会话，刷响应时间指标。
- 数学智能体输出大量模板化步骤，试图骗取过程分。

防御思路：

- 奖励分解，不让单一指标独大。
- 增加约束检查，如禁止修改测试。
- 做对抗评估，专门找“高分但作弊”的样本。
- 引入人工抽检和规则校验。

## 4. 数学推理训练案例的扩展

### GSM8K 的特点与扩展方向

GSM8K 主要是小学到初中级别的文字数学题，特点是：

- 题目短但需要多步算术推理。
- 最终答案明确，便于自动评估。
- 很适合训练“逐步分解 + 计算”能力。

若要训练更复杂数学能力，应扩展：

- 数据集：加入高等数学、证明题、竞赛题。
- 工具：允许调用符号计算器或证明检查器。
- 奖励：不只看最终答案，还看证明结构和严谨性。

### 泛化能力评估方案

要验证是否真正学会推理，而不是记住训练集，可做：

- 题型变体测试：同一知识点换表达方式。
- 分布外测试：更长链路、更复杂数字、更少提示。
- 干扰项测试：加入无关信息，看是否仍能抓住关键条件。

提升泛化能力的方法包括：

- 正则化与早停。
- 数据增强与模板多样化。
- 按难度分层训练，而不是只刷单一题型。

### 在线学习方案的关键挑战

在线学习可以让智能体在真实使用中持续收集反馈，但要解决：

- 数据质量：用户反馈噪声大，不一定可靠。
- 灾难性遗忘：新数据可能冲掉原有能力。
- 安全性：不能让恶意反馈直接污染模型。
- 发布节奏：在线更新要有灰度和回滚机制。

因此更实际的方案往往是“在线收集、离线筛选、分批更新”。

## 5. 让智能体学会工具使用

### 工具学习训练方案

给定搜索、计算器、代码执行器等工具，可把训练目标设为：

- 该用工具时会用。
- 不该用工具时不乱用。
- 选择正确工具，并传入正确参数。

奖励可拆为：

- 工具选择正确性
- 参数正确性
- 最终任务完成度
- 调用效率

### 分层强化学习方案

可把策略拆成两层：

- 高层策略：决定当前子目标，例如“先搜文档”“先跑测试”。
- 低层策略：决定具体调用哪个工具、传什么参数。

训练时，高层看任务级奖励，低层看工具级即时反馈。二者通过子目标接口衔接。

### 工具很多时的课程学习设计

一个合理课程顺序是：

1. 单工具简单任务
2. 双工具串联任务
3. 多工具但低依赖任务
4. 多工具高依赖和长链任务

进入下一阶段前，可检查：

- 当前阶段成功率是否稳定超过阈值。
- 工具误用率是否下降。
- 平均调用步数是否收敛。

本质上，课程学习是在控制探索空间，避免一开始就把 50+ 工具全部丢给策略去盲目搜索。
