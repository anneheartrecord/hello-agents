# 第十章 习题参考答案

> 参考答案仅供参考。本章最重要的，不是记住协议名字，而是理解不同协议到底在解决“Agent 与工具如何连接、Agent 与 Agent 如何协作、Agent 网络如何扩展”这三类不同问题。

## 1. MCP、A2A、ANP 的定位差异与组合方式

### 为什么三种协议强调点不同

三种协议面向的问题边界不同：

- `MCP` 强调上下文共享，本质上是在解决“模型如何以标准方式拿到工具、资源、提示模板”。
- `A2A` 强调对话式协作，本质上是在解决“多个智能体如何围绕一个任务来回沟通、分工和交付”。
- `ANP` 强调网络拓扑，本质上是在解决“当智能体数量很多时，消息如何发现、路由、扩展和容错”。

可以把它们理解为三层：

- MCP：单个 Agent 的外部能力接口层。
- A2A：多个 Agent 的协作层。
- ANP：大规模 Agent 系统的网络层。

### 智能客服系统三类需求的协议选型

对题目中的三个功能，我会这样选：

1. 访问客户数据库和订单系统：优先 `MCP`
   因为这是典型的 Agent 对工具和资源访问问题。

2. 多个专业客服智能体协作处理复杂问题：优先 `A2A`
   因为这里需要客服、售后、风控、技术支持等角色之间的任务协商和结果交接。

3. 支持大规模并发用户请求：优先 `ANP`
   因为当系统从少量智能体扩展到成百上千节点时，核心问题变成路由、发现、负载分配和故障隔离。

### 三种协议如何组合使用

一个完整的企业客服系统可以这样分层：

```text
用户请求
  -> ANP 入口路由层
  -> A2A 客服协作层
  -> MCP 工具访问层
  -> 数据库 / 订单系统 / 知识库
```

职责划分：

- `ANP` 负责把请求分发到合适的客服集群。
- `A2A` 负责让不同专业 Agent 协同完成复杂工单。
- `MCP` 负责标准化访问订单、物流、知识库、退款规则等能力。

## 2. MCP：工具、资源、提示模板与远程扩展

### 一个数据分析 MCP Server 的设计

可设计三个核心工具：

- `db_query`：执行受控 SQL 查询，只允许只读语句。
- `chart_render`：把结构化结果渲染成图表配置或图像。
- `report_generate`：结合查询结果和图表生成分析报告。

协作流程：

```text
用户问题
  -> db_query
  -> chart_render
  -> report_generate
  -> 输出分析结果
```

工程上要注意：

- 工具权限要分级。
- 查询要做 schema 白名单和行数限制。
- 图表和报告生成应保留中间产物，便于审计。

### Resources 和 Prompts 的作用

- `Resources` 适合暴露静态或半静态上下文，例如知识文档、配置、模板、说明书。
- `Prompts` 适合暴露可复用的任务模板，例如“代码审查提示词”“合同分析提示词”。
- `Tools` 适合执行动作，例如查询、写入、调用外部 API。

一个合理场景是“财务分析助手”：

- 用 `Resources` 提供会计制度、报表口径说明。
- 用 `Prompts` 提供“季度经营分析”模板。
- 用 `Tools` 执行数据库查询和图表生成。

这样 Agent 拿到的不只是动作能力，还有背景知识和任务模板。

### JSON-RPC + stdio 的优势与局限

优势：

- 协议简单，语言无关。
- 本地进程通信延迟低。
- 安全边界相对清晰，不必默认暴露到网络。

局限：

- 更适合同机或近程调用，不适合天然分布式。
- 进程管理、重启、健康检查要额外处理。
- 对浏览器、远程 SaaS、跨 VPC 场景不够友好。

如果要支持远程 MCP Server，可增加：

- `HTTP/WebSocket transport`
- 认证层，例如 token / mTLS
- 会话管理与超时控制
- 重试、断线重连、服务发现机制

## 3. A2A：协作流程、冲突解决与框架桥接

### 给“研究团队”加上 Reviewer 后怎么协作

三个角色可以这样分工：

- `Researcher`：查资料、整理论据。
- `Writer`：组织结构、撰写正文。
- `Reviewer`：评估论证质量、结构问题和表达缺陷。

协作流程：

1. Researcher 提交研究摘要。
2. Writer 生成论文草稿。
3. Reviewer 返回修改意见。
4. Writer 依据意见修订。
5. 若 Reviewer 认为达标，则交付最终版本。

这本质上把 A2A 从“双人协作”升级成“生产-审查闭环”。

### 冲突解决机制怎么设计

如果两个智能体观点冲突，不能只靠“谁先说话”决定结果。建议扩展两类消息：

- `negotiation`：双方说明立场、证据和让步条件。
- `voting`：多个 Agent 或裁判 Agent 对候选方案投票。

一个简单流程：

1. 发生冲突时进入 `negotiation`。
2. 若两轮协商后仍无共识，则进入 `voting`。
3. 若仍无法裁决，则升级到人工或规则引擎。

### A2A 与 AutoGen / CAMEL 的关系

`A2A` 更像通信协议标准，`AutoGen`、`CAMEL` 更像构建多智能体系统的框架。

它们不是简单替代关系：

- 框架负责角色管理、运行时、对话控制。
- 协议负责不同系统之间如何交换消息。

如果要和 AutoGen 互通，可设计一个 `A2ABridge`：

- 把 A2A 消息映射成 AutoGen 的内部消息对象。
- 把 AutoGen Agent 的输出再包装回 A2A 标准消息。
- 保留任务 ID、上下文、角色元数据，避免语义丢失。

## 4. ANP：拓扑、路由与容错

### 什么时候选星型、网状、分层拓扑

| 拓扑 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| 星型 | 控制简单、中心调度清晰 | 中心节点易成瓶颈和单点故障 | 小规模统一调度 |
| 网状 | 灵活、去中心化强 | 路由复杂、治理成本高 | 高自治协作网络 |
| 分层 | 可扩展性和治理性更平衡 | 设计复杂，需要层间协调 | 中大规模企业系统 |

当规模从 10 个 Agent 扩展到 1000 个 Agent 时，通常会从“星型或小范围网状”演进到“分层 + 局部网状”的混合结构。

### 智能路由算法怎么做

路由分数可综合以下因素：

```text
route_score = 0.35 * capability_match + 0.25 * current_load_inverse + 0.20 * latency_inverse + 0.20 * trust_score
```

执行流程：

1. 先根据任务标签过滤掉不具备能力的 Agent。
2. 再按路由分数排序。
3. 选择主路径与备份路径。
4. 若主路径失败，则自动切换备份。

### 关键 Agent 故障时的容错机制

可设计四层容错：

- 故障检测：心跳、超时、健康检查。
- 备份切换：主备 Agent 或同角色池化替代。
- 状态恢复：将关键上下文和任务状态持久化到共享存储。
- 流量重分配：由路由层把流量导向健康节点。

对“交通管理 Agent”这类关键节点，还应设置降级模式，例如在智能决策不可用时回退到规则控制。

## 5. 安全、隐私与信任治理

### MCP 工具调用的安全风险与权限控制

风险主要有：

- 模型误调用危险工具。
- prompt injection 诱导调用高危接口。
- 工具参数被构造成越权操作。

更稳妥的权限控制应包括：

- 工具分级：读、写、危险操作分离。
- 基于角色的授权：不同 Agent 只能访问特定工具集合。
- 参数校验与策略检查：例如删除路径必须位于沙箱目录。
- 高危工具二次确认：需要人工批准或策略引擎放行。

### A2A / ANP 的端到端加密设计

可采用：

- Agent 身份证书或公私钥对。
- 会话协商生成对称密钥。
- 消息体用对称加密，消息摘要做签名。
- 中间路由节点只看必要元数据，不解密正文。

这样可同时满足保密性、完整性和身份认证。

### 大规模网络中的信任评估系统

每个 Agent 可维护动态 `trust_score`，输入包括：

- 历史任务成功率
- 消息真实性和一致性
- 社区评价或其他 Agent 的反馈
- 是否出现恶意行为，如刷请求、异常流量、频繁提供错误结果

系统根据评分动态调整策略：

- 高信任 Agent 优先参与关键协作。
- 低信任 Agent 降低权重、限制权限，必要时隔离。

这说明在大规模 Agent 网络里，通信协议只是基础，治理系统同样是核心能力。
